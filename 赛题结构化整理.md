# AI全栈极速黑客松·赛题结构化整理

## 一、赛题概述

### 核心任务
开发一个**学科知识整合智能体**，将7本教材整合压缩到不超过原始体量30%的精华版本，且教学效果不打折。

### 比赛时长
5小时

### 核心价值
解决高校教学中多教材并行的冗余问题：
- 学生：避免重复阅读浪费时间
- 教师：便于评估教材交叉与缺失
- 管理者：获得数据驱动的课程整合依据

---

## 二、功能要求（P0 - 必须实现）

### 1. 多格式教材加载与解析

| 要求项 | 具体内容 |
|--------|----------|
| 支持格式 | PDF（必须）、Markdown（必须）、TXT（必须）、Word .docx（建议）、Excel（可选） |
| 上传方式 | 拖拽上传、点击选择、批量上传 |
| 显示信息 | 文件名、格式、大小、解析状态 |
| 输出结构 | 统一JSON格式（textbook_id, filename, title, chapters等） |
| PDF处理 | 章节标题识别、页眉页脚过滤、图表跳过 |
| 大文件策略 | 逐页解析，避免一次性加载到内存 |

**验收标准**: 上传PDF后正确识别章节结构并显示在前端

### 2. 单本教材知识图谱构建与可视化

| 要求项 | 具体内容 |
|--------|----------|
| 知识点提取 | LLM提取核心知识点（概念、定理、方法、现象） |
| 输出结构 | JSON格式（id, name, definition, category, chapter, page） |
| 关系类型 | 至少3种：前置依赖、并列关系、包含关系、应用关系 |

**验收标准**: 选择教材后生成JSON格式的节点和边列表，前端可视化展示

### 3. 知识图谱交互

| 交互功能 | 说明 |
|----------|------|
| 节点点击 | 弹出/侧边展示知识点详情 |
| 频次可视化 | 多教材频次→节点大小/颜色深度 |
| 教材来源区分 | 不同颜色标识 |
| 基本交互 | 缩放、拖拽画布、拖拽节点 |
| 搜索（建议） | 关键词高亮匹配节点 |

**推荐技术**: D3.js、ECharts、Cytoscape.js、AntV G6

### 4. 跨教材知识图谱整合（核心难点）

| 关键点 | 说明 |
|--------|------|
| 语义对齐 | 识别不同教材中同一概念的知识点 |
| 对齐算法 | Embedding语义相似度 + LLM判断（推荐结合） |
| 整合决策 | merge/keep/remove三种类型 |
| 压缩比控制 | 整合后≤原始30% |
| 可视化对比 | 整合前后图谱对比（建议） |

**验收标准**: 加载2本以上教材，自动识别重复知识点，压缩比≤30%

### 5. RAG精准问答

**完整Pipeline**:

| 步骤 | 要求 |
|------|------|
| 分块 | 500-800字，50-100字重叠，保留元数据 |
| 嵌入 | sentence-transformers或OpenAI Embedding（中文选BGE） |
| 存储 | FAISS或ChromaDB，检索top-5 |
| 生成 | 只基于上下文回答，附带来源引用 |

**前端界面**: 问答输入框 + 回答区 + 引用来源列表 + 索引状态

**API设计**:
- `POST /api/rag/index` - 建立向量索引
- `POST /api/rag/query` - 问答查询
- `GET /api/rag/status` - 查询索引状态

**加分**: 自建RAG Benchmark（20-50测试问题）

### 6. Agent架构设计与说明文档

**必须包含**:
- a) 架构总览（Mermaid图）
- b) 设计决策论证（为什么选择这种架构）
- c) 数据流与调用链路
- d) 取舍与权衡

**评分关键**: 论证深度 > Agent数量

### 7. 整合建议与多轮对话

| 功能 | 说明 |
|------|------|
| 决策解释 | 说明每项整合决策理由 |
| 对话修改 | 教师通过自然语言修改整合方案 |
| 实时更新 | 图谱实时反映修改 |
| 持久化 | 对话历史保留上下文 |

### 8. Web交互界面

**布局建议**:
- 左侧：教材管理区
- 中间：知识图谱可视化区（最大面积）
- 右侧：功能面板（Tab切换：整合/RAG问答/对话/报告）

**分辨率**: 1920×1080正常显示

### 9. 整合报告

**必须内容**:
- 整合概览（数量、字数、压缩比）
- 整合决策摘要（合并/保留/删除统计）
- 知识图谱统计（节点/关系变化）
- 重点整合案例（3-5个）
- 教学完整性说明

**存放路径**: `report/整合报告.md`

### 10. 开发文档

| 文档路径 | 内容 |
|----------|------|
| `docs/需求分析.md` | 子问题分解、粒度定义、重复判定标准等 |
| `docs/系统设计.md` | 架构图、数据流、技术选型、API |
| `docs/Agent架构说明.md` | 架构总览、设计决策、数据流、取舍 |
| `README.md` | 项目简介、环境依赖、安装、配置、启动 |

## 九、我的建议

### 1. 功能优先级策略

**核心功能链路**（必须优先完成）:
```
文件上传 → PDF解析 → 知识提取 → 图谱构建 → 跨教材整合 → RAG问答
```

**时间分配建议调整**:
- 文件解析（30分钟）- 基础，必须稳定
- 知识图谱构建（60分钟）- 核心展示功能
- 跨教材整合（45分钟）- 技术难点，关键得分点
- RAG问答（45分钟）- 核心功能
- 对话交互（30分钟）
- 文档编写（60分钟）- 重要评分项
- 部署测试（30分钟）

### 2. 技术选型建议

**简化路线**（5小时有限）:
- 后端：FastAPI + LangChain（减少Agent编排代码量）
- 前端：Vue 3 + AntV G6（图谱库成熟，文档丰富）
- 向量库：ChromaDB（最简单，无需额外服务）
- Embedding：BGE-small-zh（免费、中文效果好）
- LLM：DeepSeek API（便宜、速度快）

**不建议**:
- 多Agent架构（论证成本高，实现复杂）
- 自己训练模型（时间不够）
- 复杂的图谱算法（用Embedding相似度足够）

### 3. 文档策略

**Agent架构说明.md重点**:
- 一定要写"为什么选择单Agent"的论证
- Prompt工程要详细（few-shot示例、格式约束）
- RAG设计要有分块策略说明
- 局限性要诚实写（评委看重深度）

**需求分析.md关键**:
- 知识点粒度定义（怎么判定一个知识点）
- 重复判定标准（相似度阈值设定理由）
- 压缩比计算方式

### 4. 创新加分策略

**推荐低成本高收益创新**:
- 自建20个RAG测试问题（容易做，评分加分明显）
- Token消耗统计（简单实现）
- 图谱搜索功能（交互加分）

**不推荐高风险创新**:
- 多Agent系统（时间成本高，论证要求高）
- 自定义对齐算法（技术风险大）

### 5. 部署建议

- 魔搭创空间：免费、支持Gradio、国内访问快
- 提前测试部署（不要最后10分钟才部署）

### 6. 风险规避

**必须避免**:
- PDF解析不稳定（多测试几个PDF）
- 大文件OOM（必须逐页解析）
- 压缩比>30%（核心验收标准）
- 部署链接失效（评审时打不开直接淘汰）
